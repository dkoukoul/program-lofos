# program.lofos.gr — UX/UI Κατευθυντήριες Γραμμές

> Προαπαιτούμενα: [purpose-and-scope.md](./purpose-and-scope.md), [architecture-and-tech-stack.md](./architecture-and-tech-stack.md).
> Εδώ ορίζουμε κανόνες UX/UI για δύο εντελώς διαφορετικά κοινά: (α) βαθμοφόρους/επιτελείο στο διαχειριστικό, (β) επισκέπτες (γονείς/παιδιά) στη δημόσια προβολή. Δεσμευτικό, όπως τα δύο βασικά docs.

## 1. Γενικές αρχές (και για τα δύο κοινά)

- **Mobile-first, responsive**: κάθε οθόνη σχεδιάζεται πρώτα για κινητό, μετά επεκτείνεται σε μεγαλύτερες οθόνες. Single-column layout σε κινητό· πολλαπλές στήλες μόνο όπου βοηθάει σε ευρύτερες οθόνες.
- **Απλότητα πάνω από πληρότητα**: λιγότερα, μεγαλύτερα, σαφή στοιχεία. Καμία πυκνή φόρμα με πολλά πεδία μαζί.
- **Καμία client-side framework**: htmx partial swaps + plain CSS custom properties, όπως ορίζει το architecture doc — καμία SPA λογική, κανένα build step.
- Tap targets ≥44px, καμία οριζόντια κύλιση, native form controls (date/time pickers κινητού) όπου εφαρμόζεται.
- **Breakpoint mobile→desktop**: ενδεικτικά στα 768px (π.χ. η λίστα προγράμματος στο διαχειριστικό περνάει από 1 σε 2 στήλες πάνω από αυτό το πλάτος).
- **Ώρα πάντα σε μορφή 24ώρου** (π.χ. `18:00`, ποτέ "6:00 μμ") — σε διαχειριστικό και δημόσια προβολή.
- Όπου υπάρχει επιλογή ημερομηνίας ή ώρας, χρησιμοποιείται πάντα αντίστοιχο **date picker** και **time picker** (native `input type="date"` / `type="time"`) — ποτέ ελεύθερο πεδίο κειμένου για ημερομηνία/ώρα.
  > Σημείωση υλοποίησης: το native `type="time"` widget μπορεί να ακολουθεί τη ρύθμιση locale της συσκευής (π.χ. κάποιες συσκευές δείχνουν AM/PM μέσα στο ίδιο το picker) — αυτό είναι εκτός ελέγχου της εφαρμογής. Ο κανόνας 24ώρου είναι δεσμευτικός για **όλες** τις άλλες εμφανίσεις ώρας (κάρτες, λίστες, emails, iCal) — αυτές πάντα formatted server-side σε 24ωρη μορφή.

## 2. Διαχειριστικό (βαθμοφόροι/επιτελείο)

### 2.1 Wizard αντί για μεγάλη φόρμα

Η δημιουργία/επεξεργασία δράσης γίνεται πάντα μέσω **βηματικού wizard**, ποτέ μέσω μονολιθικής φόρμας με όλα τα πεδία μαζί. Ροή:

1. **Ημερομηνία** — date picker· default: η επόμενη διαθέσιμη Κυριακή εντός της περιόδου (η πρώτη Κυριακή χωρίς ήδη καταχωρημένη δράση ή "Χωρίς δράση"). Αν επιλεγεί ημερομηνία που έχει ήδη δράση (ίδιου τμήματος ή Συστήματος), εμφανίζεται αμέσως προειδοποίηση· η ακριβής συμπεριφορά (αποκλεισμός ή soft-warning με δυνατότητα συνέχειας) εξαρτάται από το env var `ALLOW_ACTIVITY_OVERLAP` (default: αποκλεισμός) — βλ. [architecture-and-tech-stack.md §6](./architecture-and-tech-stack.md).
2. **Τύπος δράσης** — **Τυπική** / **Ημερήσια** / **Διήμερη** / **Άλλη** / **Χωρίς δράση**. Το επιτελείο βλέπει εδώ επιπλέον ένα toggle "Δράση Συστήματος" (μη ορατό σε βαθμοφόρους τμήματος).
   - Αν επιλεγεί **Χωρίς δράση**, ο wizard πάει κατευθείαν στην Επιβεβαίωση — δεν χρειάζεται κανένα άλλο πεδίο.
   - Για **Τυπική**, προσυμπληρώνονται: ώρα έναρξης `11:00`, ώρα λήξης `13:00`, τόπος "Λόφος", τι να κρατάνε "παγούρι".
   - Για **Άλλη**, προσυμπληρώνεται μόνο το "τι να κρατάνε" = "πλήρης προσκοπική στολή" — τα υπόλοιπα πεδία μένουν κενά προς συμπλήρωση.
   - Για **Ημερήσια** / **Διήμερη**, δεν υπάρχουν ακόμα defaults (open point, purpose doc §8) — τα πεδία ξεκινούν κενά.
3. **Τόπος & ώρες** — προσυμπληρωμένα όπου υπάρχει default (βήμα 2), πάντα ελεύθερα επεξεργάσιμα μέσω date/time picker σε 24ωρη μορφή (§1).
4. **Προαιρετικά** — κόστος συμμετοχής, τι να κρατάνε (αν διαφορετικό από το default), συμμετέχοντες βαθμοφόροι.
5. **Δυναμικά πεδία** — μόνο αν χρειάζεται, προαιρετικό βήμα.
6. **Επιβεβαίωση / Αποθήκευση**.

Κανόνες:
- Ο χρήστης πρέπει να μπορεί να αποθηκεύσει αμέσως μετά το βήμα 2 αν τα defaults καλύπτουν την περίπτωση (η συνηθισμένη Τυπική δράση) — τα επόμενα βήματα είναι collapsible/optional, όχι υποχρεωτικά κλικ-κλικ-κλικ.
- Δράσεις προστίθενται **μία-μία** (όχι bulk import/CSV), σύμφωνα με το purpose doc §5.1.
- Κάθε βήμα κρατά 1–3 πεδία, όχι περισσότερα.
- Μεταβάσεις βημάτων μέσω htmx (partial swap), χωρίς πλήρες page reload.

### 2.2 Λίστα προγράμματος

- Οι δράσεις μιας περιόδου εμφανίζονται ως λίστα καρτών σε **χρονολογική σειρά** (auto-sort βάσει ημερομηνίας/ώρας, όχι χειροκίνητο drag-and-drop).
- Κάθε κάρτα δείχνει σε πρώτο επίπεδο μόνο τα ουσιώδη (ημέρα/ώρα, τύπος, τόπος)· κόστος/τι-να-φέρουν εμφανίζονται κατόπιν expand/κλικ.
- Δράσεις τύπου "Χωρίς δράση" εμφανίζονται οπτικά διαφοροποιημένες (π.χ. γκριζαρισμένη κάρτα) ώστε να ξεχωρίζουν αμέσως από κανονικές δράσεις.
- Η κατάσταση προγράμματος (Πρόχειρο/Δημοσιευμένο) είναι πάντα ορατή, με ένα κύριο, ευδιάκριτο κουμπί ενέργειας ("Δημοσίευση") — όχι θαμμένο σε menu.
- Μετά τη δημοσίευση, κάθε πεδίο που άλλαξε πρέπει να επισημαίνεται και στο διαχειριστικό (ίδια λογική με τη δημόσια σελίδα, purpose doc §5.4), ώστε ο βαθμοφόρος να βλέπει τι ακριβώς θα δει ο επισκέπτης.

### 2.3 Confirmations & validation

- Confirmation dialog μόνο σε δύσκολα αναστρέψιμες πράξεις (π.χ. διαγραφή δράσης) — όχι σε κάθε αποθήκευση.
- Μηνύματα validation (Zod) πάντα inline, δίπλα στο συγκεκριμένο πεδίο, στα ελληνικά.

## 3. Δημόσια προβολή (επισκέπτες)

### 3.1 Κοινά σε όλα τα τμήματα

- Η τρέχουσα/ενεργή περίοδος εμφανίζεται πάντα πρώτη· το ιστορικό είναι προσβάσιμο μέσω απλού selector (π.χ. dropdown μηνών), όχι κρυμμένο σε βαθύ μενού.
- Κάθε δράση = κάρτα με: ημέρα/ημερομηνία σε πρώτο πλάνο (σάρωση οπτικά ανά μέρα), ώρα, τύπος (με εικονίδιο), τόπος. Κόστος/"τι να φέρουν" εμφανίζονται **μόνο αν υπάρχουν** — ποτέ κενά πεδία.
- Οι Δράσεις Συστήματος διακρίνονται από τις δράσεις τμήματος με badge που συνδυάζει **και** εικονίδιο **και** κείμενο (π.χ. "Δράση Συστήματος"), όχι μόνο χρώμα/accent — ώστε να είναι σαφές και σε χρήστες με αχρωματοψία γιατί εμφανίζονται ίδιες σε όλα τα τμήματα.
- Πεδία που άλλαξαν μετά τη δημοσίευση επισημαίνονται έντονα (χρώμα + εικονίδιο, όχι απλά ένα αστεράκι) — purpose doc §5.4.
- Ημερομηνίες που έχουν δηλωθεί ρητά ως "Χωρίς δράση" εμφανίζονται με ξεκάθαρη ένδειξη (π.χ. "Δεν θα γίνει συγκέντρωση αυτή την Κυριακή" σε γκριζαρισμένη κάρτα) — διαφορετική από απλή απουσία δημοσιευμένου προγράμματος.
- Κουμπί εγγραφής σε iCal ευδιάκριτο, με απλή εξήγηση σε απλή γλώσσα για μη τεχνικό κοινό (π.χ. "πρόσθεσε το πρόγραμμα στο ημερολόγιο του κινητού σου").
- Ελάχιστο JS πέρα από τη βασική interactivity (π.χ. επιλογή περιόδου)· η σελίδα πρέπει να είναι γρήγορη σε αργή σύνδεση κινητού.

### 3.2 Template ανά τμήμα (ηλικιακά προσαρμοσμένο)

Και τα τρία templates μοιράζονται το ίδιο data contract (ίδια πεδία, ίδια λογική badges Συστήματος/αλλαγών) και τη βασική δομή σελίδας· διαφέρουν σε παλέτα, εικονογράφηση/iconography, τυπογραφία, και — μόνο για την Αγέλη κατά κανόνα — ένα προαιρετικό "θέμα περιόδου".

| Τμήμα | Ηλικίες | Ύφος | Ειδικά στοιχεία |
|---|---|---|---|
| **Αγέλη** | 6–10 | Παιχνιδιάρικο, ζωηρό, εικονογραφημένο | Προαιρετικό μηνιαίο θέμα/μύθος (π.χ. "Ο Μόγλης", Minions) — τίτλος θέματος + hero εικόνα/mascot σε προκαθορισμένο σημείο· στρογγυλεμένες κάρτες, μεγαλύτερη γραμματοσειρά, απλά εικονίδια ανά τύπο δράσης |
| **Ομάδα** | 10–14 | Περιπετειώδες, προσκοπικό | Παλέτα γης/khaki, εικονίδια τύπου badge/patch (πεζοπορία, κατασκήνωση, παιχνίδι), αίσθηση "ημερολογίου περιπέτειας" |
| **Κοινότητα** | 14–18 | Μοντέρνο, απλό, καθαρό | Ουδέτερη/minimal παλέτα, τυπογραφικά-οδηγούμενο σχέδιο, καμία παιδική/cartoon αισθητική — σαν καθαρό agenda/event listing |

- Η επιλογή template καθορίζεται από τον τύπο τμήματος (σταθερή στον κώδικα, όχι configurable data) — μόνο το προαιρετικό "θέμα περιόδου" (τίτλος + εικόνα, `programs.theme_title`) είναι επεξεργάσιμο ανά περίοδο, κυρίως για την Αγέλη αλλά τεχνικά διαθέσιμο σε όλα τα τμήματα.
- Υλοποίηση (βλ. [architecture-and-tech-stack.md §3, §8](./architecture-and-tech-stack.md)): τρία section-specific JSX component variants (`views/public/templates/agele.tsx`, `omada.tsx`, `koinotita.tsx`) πάνω στο ίδιο shared layout (`views/public/layout.tsx`) — CSS custom properties + ένα-δύο section-specific components (hero/header, icon set), όχι ξεχωριστά, ασύνδετα codebases ανά τμήμα.
